cap = max in-memory size  ·  path = on-disk root  ·  warm read = median hit latency

Pick by lifetime, not by raw latency. The memory backend wins every read but loses the cache on exit; choose disk or remote the moment the cache needs to outlive the process. Speed is the easy axis — durability is the one that actually decides it.

Ready-to-paste commands

# memory — fastest, ephemeral (the demo default)
forgecache build --store mem --cap 512MB

# disk — survives restarts, tiny footprint
forgecache build --store disk --path ".fc/"

# remote — shared team cache
forgecache build --store s3 --bucket "demo-cache" --prefix teamA

How a read resolves

Caveat. The hoist-on-miss step doubles writes for cold reads. On a write-heavy workload the remote backend's egress cost can outweigh the read win — profile before enabling it for everyone.
Don't raise --cap to fit the whole repo. Above the working-set size the memory backend just inflates RSS for no hit-rate gain — the demo's 4 GB-cap run was no faster than 512 MB and used 8x the memory.

How to choose

  • Ephemeral CI / inner loop → memory: fastest reads, nothing to clean up.
  • Local dev across restarts → disk: durable, tiny footprint, near-memory speed.
  • Shared team cache → remote: one cache for everyone, at a cold-miss latency cost.
