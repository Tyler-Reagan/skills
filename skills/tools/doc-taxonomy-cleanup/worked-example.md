# Worked Example — Anatomy of a Cleanup

Synthetic walkthrough of a documentation-tree cleanup using fictional project names and files. The purpose is to ground the *method* in concrete shape — what the triage table looks like, what compression ratios to expect, what kinds of drift get caught — without over-fitting to any specific real cleanup.

> **Maintenance note.** All names and numbers in this document are illustrative. Do not edit this file to add a real-world case study from any specific codebase — keep examples generic and synthetic. See the corresponding anti-pattern in [SKILL.md](SKILL.md).

## Compression observed (fictional totals)

After Chunk 3 (the rewrites), the PROMOTE outcomes:

| Source plan                       | Plan lines | Decision-record lines | Compression |
|-----------------------------------|------------|------------------------|-------------|
| auth-rework.md                    | 1,100      | 60                     | ~95%        |
| analytics-pipeline-v2.md          | 900        | 110                    | ~88%        |
| webhook-replay.md                 | 700        | 90                     | ~87%        |
| inbox-redesign.md                 | 200        | 80                     | ~60%        |
| db-pool-tuning.md                 | 180        | 120                    | ~33%        |
| **Aggregate**                     | **3,080**  | **460**                | **~85%**    |

Records that compressed least (db-pool-tuning, inbox-redesign) had less plan-form scaffolding to begin with — their plans were already close to decision-record shape. Records that compressed most (auth-rework, analytics-pipeline-v2) had heavy phase plans / alternatives-considered / planning-time open questions that the rewrites cut.

**Implication for any new cleanup:** if your PROMOTE rewrites aren't compressing by at least ~60%, double-check what survived. Phase plans, detailed alternatives, planning-time open questions are the usual culprits.

## Categories of drift the rewrites fixed

This is the *shape* of drift to expect, not specific drift from a real case study. Every PROMOTE rewrite typically fixes one or more of these:

1. **Renamed symbols.** The plan refers to a function/type/method by an older name; the code has since renamed it. The rewrite reflects the current name. Catch this with `grep` on the symbol in the current tree.
2. **Rerouted control flow.** The plan describes a flow through component A → B → C; the code now goes A → D → C, with B removed entirely. The rewrite documents the actual flow.
3. **Abandoned sub-decisions.** A plan often includes multiple sub-decisions ("we'll do X *and* Y *and* Z"). Some land; some get dropped silently. The rewrite keeps only the sub-decisions still reflected in code.
4. **Pre-shipped opens.** The plan lists open questions that were resolved during implementation; the resolutions live in commits, not in the plan. The rewrite drops the resolved opens (or moves them to a brief *Decisions* row with the resolution).
5. **Multi-phase rollouts that finished.** Plans often have phase plans (`Phase 1 → Phase 2 → Phase 3`). Once shipped, the phase plan is irrelevant — it's just git history. The rewrite cuts it.
6. **Source-of-truth violations.** Plans sometimes embed code snippets that lived elsewhere in the repo at write-time but have since diverged. The rewrite drops the snippet and references the source-of-truth path.

## Mechanical follow-up (Chunk 5)

Per [SKILL.md](SKILL.md)'s chunk breakdown, code-reference repointing is non-optional. Expected work in a fictional cleanup:

- Code comments that reference `docs/plans/<old-path>.md` get repointed to `docs/decisions/<new-path>.md` (or to `docs/plans/archive/<old-path>.md` for archives — though archive references are usually cut entirely, not repointed).
- Inline `// TODO(see plan-doc-name)` comments get either repointed or resolved (if the plan was deleted, the TODO is reassessed).
- Cross-doc links within the doc tree (e.g. `architecture.md` referencing a moved plan) get fixed.

**No logic changes in this chunk.** Comment-only touches. The "smallest delta" principle applies.

## What's transferable, not specific

The takeaways from this synthetic walkthrough that apply to any cleanup:

1. **Status lines lie at scale.** When a doc tree has rotted, expect the declared statuses to be wrong on a majority of stale files. Don't read them.
2. **The compression target is severe.** Aim for ≥60% reduction on PROMOTE rewrites. Less than that, plan-form scaffolding survived.
3. **Drift is the rule, not the exception.** Expect every PROMOTE rewrite to fix at least one wrong claim about the code. If a rewrite reflects the plan verbatim, you didn't actually code-anchor it.
4. **Mechanical follow-up matters.** A rewrite that doesn't repoint references leaves dangling anchors. Chunk 5 is non-optional in any non-trivial cleanup.
5. **One coherent landing is better than dribbled changes.** Land the taxonomy + the rewrites + the repoints together. Reviewer cost is lower when the new shape lands in one piece than when it accretes across many small changes.
